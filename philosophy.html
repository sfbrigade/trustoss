Philosophy

Open source is designed to build trust

On a number of different levels, we can see this principle in action:

  • On a source code level, releaseing the source itself builds trust with your adoptors that the code won't cause any issues, as they can personally audit its functionality
  • On a licencing level, an open source license builds trust that your adoptors won't be brought into costly litigations for copying (and/or possibly redistributing) your code.
  • On a documentation level, the ideal open source project establishes trust that the application or library can be installed, how/where adoptors may contribute back to the project, and how the project may be governed.
  • On a versioning level, making sure there is a proper git tag and release log for every release builds trust with your adoptors that upgrading the software won't cause their system to begin melting before their eyes. It lets them know how and when to migrate their systems (if needed) and who to credit for the features they like.

Open Source projects are diverse

Projects that have been released into the open source ecosystem are widely diverse. They range from utilities to full-blown applications, from small blogs to massive SaaS products. Even the languages they're built in differ so widely from each other that creating a unifying standard for project architecture can appear to be an unsurmountable task, even on the smallest of projects. While TrustOSS is not designed to solve this entire problem, it does try to tackle to one portion of it that seems to be the same across the board: documentation.
